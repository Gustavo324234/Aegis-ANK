📄 [ANK-101] Especificación: Cognitive Process Control Block (PCB)
1. Visión General (Abstract)
En los sistemas operativos tradicionales (Linux, Windows), el PCB (o Task Control Block) es la estructura de datos que el Kernel usa para almacenar todo el contexto de un proceso, permitiendo pausarlo (Context Switch) y reanudarlo sin perder datos.
En el Aegis Neural Kernel (ANK), el "proceso" no es un binario compilado, sino una Misión Cognitiva delegada a un LLM (ej. "Refactorizar el módulo de base de datos"). El Cognitive PCB es la estructura JSON/Python que almacena el "estado mental" de la IA para esa tarea específica.
Objetivo del Ticket: Definir la estructura de datos pura (DataClass / Pydantic Model) que representará un proceso en la memoria del Kernel, desacoplando el estado de la tarea de la ventana de contexto del LLM.
2. Esquema de Datos (Data Schema)
Cada proceso en ANK será instanciado bajo el siguiente esquema estricto:

3. Desglose de Componentes
3.1. Identificación y Metadatos (pid, parent_pid, priority)
pid: Identificador único del proceso.
parent_pid: Crucial para el "Enjambre". Si el proceso padre ("Crear Web") necesita delegar una subtarea ("Crear CSS"), generará un proceso hijo. Si el hijo muere, el padre lo sabe.
priority: Un entero (0-10). Las peticiones directas del usuario por chat son Prioridad 10 (interrumpen lo que sea). Las tareas de fondo (ej. indexar archivos) son Prioridad 2.
3.2. Máquina de Estados (state)
El campo más importante para el Scheduler. Solo puede tener estos valores:
NEW: Proceso creado, pero el Grafo de Tareas (DAG) aún no ha sido calculado.
READY: El proceso sabe lo que tiene que hacer (tiene su L1 instruction) y está en la cola esperando que la "CPU" (el LLM) se libere.
RUNNING: El LLM está activamente generando tokens para este proceso.
WAITING_SYSCALL: El LLM pidió usar una herramienta (ej. leer un archivo o ejecutar bash) y está pausado esperando que el sistema tradicional responda.
COMPLETED: Tarea finalizada con éxito.
FAILED: La tarea superó los ciclos permitidos sin resolver el problema.
3.3. Contador de Programa (program_counter)
En código normal, el PC apunta a una línea de código. En ANK, apunta a un Nodo del DAG. Indica qué micro-tarea exacta estamos resolviendo dentro de la gran misión. Cuando el current_node se resuelve, el PC avanza al siguiente.
3.4. Gestor de Paginación (memory_pointers)
La cura para la amnesia de los modelos pequeños.
L1_instruction: El prompt inmediato. Corto, estricto, tipo ensamblador cognitivo. (ej. "Compara X con Y. Responde JSON").
L2_context_refs: URIs (Uniform Resource Identifiers) a fragmentos de texto exactos que deben ser inyectados en el prompt justo antes de inferir. El LLM no guarda el texto en el PCB, solo guarda el puntero (file://...).
swap_refs: Punteros a la base de datos vectorial (Chroma/ChromaDB/SQLite Vector) para contexto que es relevante pero no crítico.
3.5. Registros Cognitivos (registers)
Memoria de muy corto plazo (RAM volátil).
accumulator: Si la tarea es escribir 500 líneas de código, un modelo de 1B fallará. El acumulador permite ir guardando el código en "chunks" a medida que avanza el DAG.
temp_vars: Variables extraídas por el LLM en pasos anteriores que necesita en el futuro (ej. nombres de variables, rutas de red).
sys_error_traceback: Si el Execution Ring (Ring 0) detecta que el código falló, guarda el error aquí para que en el siguiente ciclo el LLM lo lea e intente auto-sanarse.
3.6. Métricas y Límites (Watchdog) (execution_metrics)
Previene bucles infinitos de alucinación.
Si cycles_executed supera max_cycles_allowed (por ejemplo, el modelo se equivocó de sintaxis 15 veces seguidas), el Kernel dispara un Cognitive Kernel Panic, aborta la tarea y le avisa al usuario humano.
4. Criterios de Aceptación (Definition of Done)
Para dar este ticket por cerrado y convertido en código, se debe:

Crear un archivo ank/core/pcb.py.

Implementar la estructura usando pydantic.BaseModel o dataclasses.dataclass con tipado estricto (Strict Typing) en Python.

Implementar un método to_json() y from_json() para serializar el estado al disco (esto permite que ANK se reinicie físicamente sin perder el hilo de lo que la IA estaba pensando).

Escribir un test unitario (tests/test_pcb.py) que cree un PCB ficticio, lo cambie de estado a WAITING_SYSCALL y verifique la inmutabilidad de los demás datos.

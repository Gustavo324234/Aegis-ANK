📄[ANK-103] Especificación: Servidor gRPC (ANK-Bridge)
1. Visión General (Abstract)
El ANK-Bridge es la capa de transporte y red del Aegis Neural Kernel. Al estar escrito en Rust, operará como un servidor de alto rendimiento capaz de manejar miles de peticiones simultáneas con una latencia de red mínima.
Su función principal es exponer las capacidades cognitivas del Kernel al exterior (Aegis UI, scripts, clientes externos) mediante el protocolo gRPC sobre HTTP/2, serializando los datos con Protocol Buffers.
Objetivo del Ticket: Implementar el servidor gRPC utilizando el crate tonic, conectar las llamadas de red entrantes al Cognitive Scheduler (vía canales MPSC) y establecer un sistema de Server-Side Streaming para transmitir el "pensamiento" de la IA en tiempo real.
2. Arquitectura de Red y Conectividad
El servidor debe configurarse con los siguientes parámetros técnicos:
Protocolo: gRPC (HTTP/2).
Puerto por defecto: 50051 (configurable vía variable de entorno ANK_PORT).
Interfaz (Bind): 127.0.0.1 (loopback) por motivos de seguridad en despliegues locales, permitiendo configuración a 0.0.0.0 para despliegues distribuidos (Swarm/Nodos).
Concurrencia: Gestionado automáticamente por el runtime asíncrono tokio.
3. Implementación del Servicio (KernelService)
El código generado a partir de kernel.proto creará un trait KernelService. El servidor debe implementar este trait con las siguientes lógicas para cada endpoint:
A. Endpoint: SubmitTask (Unario)
Misión: Ingestar una nueva orden al Kernel.
Input: TaskRequest (Contiene el prompt, la prioridad y la preferencia de modelo).
Flujo Interno:
Validar la petición (asegurarse de que el prompt no esté vacío).
Generar un PID único (usando uuid::Uuid::new_v4()).
Construir un PCB en estado NEW.
Enviar el PCB al canal emisor (Producer) del Cognitive Scheduler.
Output: TaskResponse con el PID generado y confirmación (accepted: true).
Latencia Objetivo: < 5 milisegundos (No bloquea esperando a la IA).
B. Endpoint: WatchTask (Server-Side Streaming)
Misión: Suscripción en tiempo real a los eventos de un proceso específico.
Input: TaskSubscription (Contiene el PID a observar).
Flujo Interno:
El servidor recibe la petición y crea un canal de stream (ej. tokio::sync::mpsc::channel dedicado para esta conexión).
Registra este canal en el Event Broker del Kernel (un publicador interno que mapea PIDs a canales de clientes).
El método entra en un bucle asíncrono, leyendo del canal interno y cediendo (yield) objetos TaskEvent al cliente gRPC.
Cierra el stream automáticamente si recibe un evento de estado COMPLETED o FAILED desde el Kernel.
Output: Stream<TaskEvent> (Flujo continuo de tokens de pensamiento, resultados de Syscalls y cambios en el PCB).
C. Endpoint: GetSystemStatus (Unario)
Misión: Telemetría en tiempo real.
Input: Empty.
Flujo Interno:
Consultar a través del cHAL (Hardware Abstraction Layer) qué modelo está cargado en VRAM.
(Opcional en esta fase) Consultar crates de sistema (ej. sysinfo) para obtener el % de CPU y RAM.
Output: SystemStatus (Estructura con métricas de hardware y estado del modelo).
4. El "Event Broker" (Mapeo de Streams)
Para que WatchTask funcione, el servidor necesita un mecanismo para saber qué cliente está escuchando qué proceso.
Estructura Sugerida: Arc<RwLock<HashMap<String, mpsc::Sender<TaskEvent>>>>
La clave (String) es el PID.
El valor es el emisor del canal conectado al cliente gRPC.
Cuando el Scheduler o el LLM generan un token nuevo para un PID, lo envían a este Broker, quien lo enruta al cliente correspondiente (si hay alguno suscrito).
5. Criterios de Aceptación (Definition of Done)
Para considerar este ticket completado y probado, se debe:

Crear src/server/mod.rs (o similar) en el crate ank-server.

Configurar el constructor Server::builder() de tonic para escuchar en el puerto designado.

Implementar la estructura AnkRpcServer que herede del trait gRPC generado.

Inyectar correctamente el canal de envío (Sender) hacia el Cognitive Scheduler dentro del estado del servidor.

Test de Integración (gRPC Client):
Levantar el servidor en un puerto temporal.
Crear un cliente gRPC dummy (en el mismo test de Rust) que llame a SubmitTask.
Verificar que el servidor devuelve un UUID válido y no se bloquea.

(Opcional pero recomendado) Configurar tracing para emitir logs en consola cada vez que se recibe una conexión gRPC (ej. INFO: Incoming SubmitTask from 127.0.0.1).
Notas de Implementación
Dependencias Clave: tonic, prost, tokio, tokio-stream.
Manejo de Errores: Devolver siempre códigos de estado estándar de gRPC (tonic::Status::invalid_argument, tonic::Status::internal) en lugar de hacer panic!() en el hilo del servidor.
